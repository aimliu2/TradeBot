# 2026-03-04 — ML Data Preparation Thought Experiment

## Dataset
- BTCUSD 15m candles, 70,930 rows
- Source: `data/mlData/BTCUSD-15m-raw.jsonl`

## Features under review

### isSTR / isITR / isLTR
- Categorical signals: -1, 0, 1
- Class imbalance:
  - isSTR: ~15% non-zero (acceptable)
  - isITR: ~5% non-zero (borderline)
  - isLTR: ~1.6% non-zero (rare — ~1,169 signal rows)
- For a **sequence model**, input sparsity is less of a problem than tabular.
  The hidden state can gate dramatically on rare signals.
- Real concern is **sample efficiency**: ~818 isLTR signal events in training split.

### hiDTK / lowDTK / closeDTK
- Percentage deviation of OHLC from `LTRkeylv` (last swing key level).
- `LTRkeylv` is static **98.4% of the time**, with runs up to 328 bars (avg 60 bars = ~15 hrs).
- Risk: DTK values accumulate drift silently — same DTK number means different things at bar 3 vs bar 300 of a run.
- Outliers (±20%) come from very long stale runs.
- **Mitigation**: add `barsSinceLTR` feature to give the model staleness context.
- Additional risk: when isLTR fires, DTK resets to ~0, so DTK may encode the signal implicitly — isLTR could be redundant.

### Y (target)
- Binary direction of next candle: close > open → 1, else -1.
- Nearly balanced: ~35,530 up / ~35,373 down / 26 neutral.
- **Bug found and fixed**: original code used `shift(1)` (predicting the past).
  Corrected to `shift(-1)` to predict the next candle.

## Key finding: isLTR signal quality
After fixing the Y shift, the isLTR→Y relationship is strongly inverse:

```
isLTR =  1 (swing HIGH): Y=+1: 114,  Y=-1: 470  (~80% reversal down)
isLTR = -1 (swing LOW):  Y=+1: 482,  Y=-1: 103  (~82% reversal up)
```

- When a swing high is confirmed, the next candle tends to reverse downward.
- When a swing low is confirmed, the next candle tends to reverse upward.
- This aligns with SMC exhaustion/retracement logic.
- Despite rarity, isLTR is a **high-quality feature** with strong predictive power.

## Signal quality confirmed: isSTR and isITR

All three signals share the same strongly inverse ~80-82% pattern:

```
isSTR = +1: Y=-1: 4383,  Y=+1: 1083  (~80% reversal down)
isSTR = -1: Y=-1: 1060,  Y=+1: 4352  (~80% reversal up)

isITR = +1: Y=-1: 1471,  Y=+1:  350  (~81% reversal down)
isITR = -1: Y=-1:  331,  Y=+1: 1491  (~82% reversal up)

isLTR = +1: Y=-1:  470,  Y=+1:  114  (~80% reversal down)
isLTR = -1: Y=-1:  103,  Y=+1:  482  (~82% reversal up)
```

### Implications
- All three signals are **genuine and high quality** — 80% single-feature accuracy before any model is exceptional.
- Pattern is **consistent across timeframes** — STR/ITR/LTR all encode exhaustion/reversal logic at different scales.
- Signals likely **compound** — when multiple fire in the same direction, reversal is probably stronger. To verify:
  ```python
  df['signal_sum'] = df['isSTR'] + df['isITR'] + df['isLTR']
  df[df['signal_sum'].abs() > 1].groupby(['signal_sum', 'Y']).size()
  ```
- Features may be **partially redundant** — overlapping ~80% pattern, but confluence value justifies keeping all three.

## Normalization discussion: DTK vs StandardScaler

Two types of normalization serve different purposes:

| Normalization | What it solves | Needed? |
|---|---|---|
| DTK formula `(price - keylv) / keylv` | Price-level dependency across time (BTC $50k vs $100k) | Yes, keep it |
| StandardScaler on DTK | Scale imbalance vs other features during training | Yes, on top |
| StandardScaler on `barsSinceLTR` | Wildly different range (1–328) | Especially yes |
| StandardScaler on `isSTR/isITR/isLTR` | Already in [-1, 1] | Optional, marginal benefit |

### Key insight
- DTK formula is **domain normalization** — makes financial sense (removes absolute price scale).
- StandardScaler is **statistical normalization** — helps the optimizer treat all features equally.
- They solve different problems. DTK does NOT replace StandardScaler.
- Standardization is a linear, reversible transformation — **no information is lost**.

### Scale mismatch without StandardScaler
```
isSTR / isITR / isLTR : magnitude ~1.0
closeDTK              : mean ~0.003, std ~0.028
barsSinceLTR          : range 1–328
```
The optimizer disproportionately follows gradients from large-scale features and underweights small-scale ones.

### Important: fit scaler on training data only
```python
scaler = StandardScaler()
train_scaled = scaler.fit_transform(train_df[dtk_cols])  # fit on train only
val_scaled   = scaler.transform(val_df[dtk_cols])        # transform only, no refit
# Fitting on full dataset = data leakage
```

## TODO: ITRkeylv vs LTRkeylv as DTK reference

### Question
Would switching from LTRkeylv to ITRkeylv as the DTK reference improve data quality?

### Trade-off

| | LTRkeylv (current) | ITRkeylv |
|---|---|---|
| Update frequency | ~60 bars avg (max 328) | ~20 bars avg |
| Staleness | High | Medium |
| Structural significance | Major swing H/L | Medium-term swing H/L |
| DTK drift risk | High (±20% outliers) | Lower |

### Key insight
- ITRkeylv updates ~3x more frequently → less stale, tighter DTK values
- But loses structural weight — ITR levels are less significant than LTR levels
- For **next 15m candle prediction**, shorter-term levels may be MORE relevant than LTR

### Suggested approach: use both as separate features
```python
# Two DTK reference sets — additive, not redundant
df["hiDTK_ITR"]    = (df["high"]  - df["ITRkeylv"]) / df["ITRkeylv"]
df["closeDTK_ITR"] = (df["close"] - df["ITRkeylv"]) / df["ITRkeylv"]

df["hiDTK_LTR"]    = (df["high"]  - df["LTRkeylv"]) / df["LTRkeylv"]
df["closeDTK_LTR"] = (df["close"] - df["LTRkeylv"]) / df["LTRkeylv"]
```
- LTR features: "where is price relative to major structure"
- ITR features: "where is price relative to recent structure"
- Sequence model can learn to weight them contextually

### Also consider: STRkeylv
- isSTR fires every ~7 bars → most fresh reference
- May be the most directly relevant for single next-candle prediction
- Worth experimenting: STR, ITR, LTR DTK sets as three parallel reference frames

## barsSince features: which levels need them?

The staleness problem that motivated `barsSinceLTR` is specific to **long stale runs** causing silent DTK drift.

| Level | Avg update | Staleness problem? | Need `barsSince`? |
|---|---|---|---|
| LTR | ~60 bars (max ~328) | Severe | Yes |
| ITR | ~20 bars | Moderate | Yes, if using ITR DTK features |
| STR | ~7 bars | Minimal | No |

- `barsSinceSTR` is not worth adding — STR resets so frequently that DTK values can't accumulate meaningful drift.
- The feature would have near-constant low variance and the optimizer would learn to ignore it.
- Only add `barsSinceITR` when ITR DTK features (`hiDTK_ITR`, `closeDTK_ITR`) are actually added.
